# Mibu 專案總覽 - NotebookLM 心智圖資料

## 專案定位

Mibu 是一個「遊戲化旅遊安全平台」，核心價值是：

> **「讓每位旅客在陌生城市都有一個在地朋友」**

透過兩大模組實現：
1. **行程扭蛋模組** - 用遊戲化機制吸引旅客，AI 生成行程 + 優惠券驅動出遊
2. **旅程策劃模組** - 將旅客與目的地「在地旅程策劃師」配對，提供旅途安全與在地連結

# 一、商業模式

## 核心價值主張

| 對象 | 價值主張 |
|------|---------|
| **旅客** | 出國旅行時，有一個當地朋友隨時待命 |
| **Specialist** | 用在地知識換取收入，彈性工作 |
| **商家** | 獲得精準的旅客導流 |

## 兩大模組的關係

```
【行程扭蛋模組】              【旅程策劃模組】
   (獲客引擎)                   (核心價值)

┌─────────────┐            ┌─────────────────┐
│ AI 生成行程  │            │ 在地旅程策劃師   │
│ + 優惠券驅動 │ ─────────→ │ (數位好友)      │
│ + 遊戲化黏著 │            │                 │
└─────────────┘            │ • 線上諮詢      │
      │                     │ • 線下安全協助  │
      ↓                     │ • 旅遊相關服務  │
  吸引用戶來玩              └─────────────────┘
                                   │
                                   ↓
                         建立旅客與目的地的關係
```

## Specialist 服務模式

| 傳統導遊 | Mibu Specialist |
|---------|-----------------|
| 現場陪同 | **線上待命** |
| 按行程收費 | **按天收費** |
| 只管導覽 | **全方位協助**（含緊急狀況）|
| 旅行社派遣 | **在地素人招募** |

### 服務內容
- **線上諮詢**：行程建議、餐廳推薦、交通指引
- **緊急協助**：語言不通時幫忙溝通、出事時知道找誰
- **在地智慧**：哪裡會被坑、當地人才知道的秘密景點

## 收入來源

| 優先級 | 收入類型 | 付費方 | 模式 | 說明 |
|--------|---------|--------|------|------|

| **P1** | 優惠券核銷 | 商家 | CPS 抽佣 | 按成交付費 |

# 二、五大頁面功能拆解

## 1. 登入頁 (LoginScreen)

### 功能清單
- Apple Sign-In 一鍵登入
- Google Native 一鍵登入
- OAuth 統一認證流程
- 多身份綁定（同一帳號可綁 Apple + Google）
- Token 自動管理（7天有效期）

### 設計理念
| 設計決策 | 為什麼這麼做 |
|---------|-------------|
| 只用 OAuth，不做帳密註冊 | 降低註冊門檻，減少流失 |
| 支援多身份綁定 | 防止用戶忘記當初用哪個登入 |
| 7天 Token 有效期 | 平衡安全性與便利性 |

### 設計巧思
- **無密碼設計**：現代用戶討厭記密碼，OAuth 一鍵搞定
- **靜默續期**：Token 快過期時自動刷新，用戶無感

## 3. 扭蛋頁 (GachaScreen)

### 功能清單
- 國家/城市選擇器
- 扭蛋次數調整（1-15次，預設7）
- AI 行程生成（1-2分鐘處理）
- 優惠券獲得動畫
- 稀有度顯示（N/R/SR/SSR/SP）
- 剩餘額度提示（36次/日）
- 解鎖全球地圖入口

### 設計理念
| 設計決策 | 為什麼這麼做 |
|---------|-------------|
| 每日 36 次限制 | 製造稀缺性，讓用戶每天回來 |
| 扭蛋機視覺 | 遊戲化體驗，製造期待感 |
| AI 生成而非隨機 | 差異化，行程有邏輯不是亂湊 |
| 稀有度分級 | 讓優惠券有價值感 |

### 設計巧思
- **扭蛋動畫**：轉動 → 停止 → 揭曉，製造懸念
- **優惠券彈出動畫**：SSR/SP 有特效，增加獲得感
- **背包容量 30 格**：製造「滿了要清」的消費動力
- **城市選擇 footer 提示**：「我們正在增加更多國家」→ 眾籌入口

### 商業邏輯
```
每日限額 36 次 → 每天都要來 → DAU 提升
稀有度設計 → SSR 優惠券更有價值 → 商家可定價
背包限制 → 要去用掉或刪除 → 驅動核銷
```

### 關鍵指標
| 指標 | 數值 | 設計原因 |
|------|------|---------|
| 每日扭蛋限額 | 36 次 | 每天玩 5 分鐘差不多用完 |
| 單次抽取 | 1-15 個 | 給用戶選擇權 |
| 背包容量 | 30 格 | 約可存 2-4 次抽獎結果 |

| 設計決策 | 為什麼這麼做 |
|---------|-------------|
| 圖鑑收集概念 | 收集癖激勵探索，「集滿」的成就感 |

| 社群投票 | UGC 品質控管，爛景點會被投掉 |

### 設計巧思
- **紅點提示**：未讀的新景點，吸引用戶點進去看

- **城市分類**：方便規劃「下次去東京要去這幾個」

### 商業邏輯
```
圖鑑完成度 → 成就徽章 → 炫耀社群 → 病毒傳播
```

# 四、四大用戶角色

## 1. 旅客 (Traveler)
- **權限**：扭蛋、圖鑑、行程規劃、聊天、付費使用 Specialist
- **狀態**：立即可用
- **核心價值**：發現景點、獲得優惠、旅途安心

## 2. 商家 (Merchant)
- **權限**：優惠券發行、地點管理、交易核銷、數據分析
- **狀態**：需審核通過
- **核心價值**：獲取客流、精準行銷、ROI 追蹤

## 3. 專員 (Specialist)
- **權限**：旅客配對、聊天服務、行程協作、收入提現
- **狀態**：需審核通過
- **核心價值**：用在地知識賺取收入、彈性工作

## 4. 管理員 (Admin)
- **權限**：公告管理、黑名單管理、系統設定、數據監控
- **狀態**：超管限定
- **核心價值**：平台營運、內容審核、品質控管

# 五、推薦系統設計

## 推薦碼機制

### 功能設計
| 功能 | 說明 |
|------|------|
| 推薦碼生成 | 每用戶獨立推薦碼（G 開頭=Google，A 開頭=Apple）|
| 獎勵機制 | 邀請成功雙方各得 50 XP |
| 里程碑獎勵 | 1/3/5/10 人達成額外獎勵 |
| 排行榜 | 週榜/月榜激勵競爭 |

### 設計巧思
- **推薦碼帶登入來源**：G/A 開頭，追蹤用戶從哪來
- **雙向獎勵**：邀請人和被邀請人都有好處，降低分享心理門檻
- **排行榜**：製造競爭氛圍，重度用戶會瘋狂邀請

### 商業邏輯
```
推薦獎勵 = 獲客成本 (CAC)
用戶帶用戶 → 比廣告便宜 → 降低整體 CAC
排行榜 → 激勵 KOL 分享 → 病毒傳播
```

# 八、推進優先級

- 重點：扭蛋體驗優化、推薦系統完善

## Phase 2：驗證 Specialist PMF
- 目標：驗證「旅客願意為在地朋友付費」
- 重點：招募 5-10 位台灣 Specialist、定價測試
